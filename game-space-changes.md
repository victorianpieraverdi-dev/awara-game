# AWARA — единый документ замен для большого игрового пространства

Дата проверки: 2026-05-16 UTC

## 1. Назначение документа

Этот документ фиксирует вторую проверку наличия правок из заметок Pavel'а о большом игровом пространстве AWARA.

Задача проверки:

- понять, какие правки уже находятся в проекте;
- отделить уже реализованное от будущей архитектуры;
- собрать единый документ замен, чтобы дальше не держать эти решения только в переписке и отдельных фрагментах;
- не менять канон игры и не внедрять новые механики без отдельной задачи.

Главная формула замены:

```text
перегруженная карта всего мироздания
→ слоистая архитектура опыта
→ чистый мобильный фокус
→ панели памяти и прогресса
→ большая настольная / космическая игра как поздний уровень
```

## 5. Подтверждение по архитектурной документации

### 5.1. Эволюционные уровни игрока

В `docs/awara-living-game-architecture.md` есть раздел `Evolutionary Levels of the Player`.

Там зафиксирована лестница:

```text
Ahamkara / Mind
→ Soul / Feeling
→ Jiva / Earth and Star Temple
→ Spirit / Cosmos and Service
→ Sobor / Galactic and Universal Architecture
→ Divine Board Game
```

Смысл замены:

```text
одна плоская вселенная
→ путь развития сознания через уровни
```

Каждый уровень имеет свой масштаб:

| Уровень | Масштаб | Игровой смысл |
|---|---|---|
| Ahamkara | ум, культура, ключи | игрок учится языкам восприятия |
| Soul | тело, чувство, внутренние состояния | игрок переходит от знания к чувствованию |
| Jiva | земля, храм, пространство | игрок строит личное пространство и звёздный храм |
| Spirit | космос, род, служение | игрок несёт ответственность за дом, землю, род, союз |
| Sobor | локи, соборы, творцы | игрок работает с высокой архитектурой и коллективными структурами |
| Divine Board Game | весь путь сверху | весь мир AWARA становится большой игрой |

### 5.2. Карты, ключи, редкость и глубина

В документации найден раздел `Cards, Keys, Rarity and Depth`.

Смысл замены:

```text
карта как случайная награда
→ карта как кристаллизованный опыт
```

Рарность должна зависеть от:

```text
rarity = depth of experience + complexity of key + level of connection + amount of light + responsibility of use
```

Это соответствует заметкам Pavel'а: высокие карты приходят чаще не потому, что система стала щедрее, а потому что игрок дорос до способности нести их смысл.

### 5.3. Светообмен

В документации найден раздел `Light Exchange`.

Смысл замены:

```text
рынок / торговля
→ поле резонанса, служения и обмена возможностями
```

Светообмен должен хранить:

- что игрок может предложить;
- что игрок ищет;
- что игрок строит;
- какой обмен готов нести;
- какое служение может проявить.

Это важно: светообмен не должен стать спекулятивной экономикой.

### 5.4. Сиддхи, аура и тороид

В документации найден раздел `Siddhi, Aura and Toroid`.

Смысл замены:

```text
просто число света
→ сила поля игрока, его ауры и тороида
```

Влияния тороида описаны как символическая игровая модель:

- притяжение событий;
- уроки и испытания;
- энергетическая защита;
- способность держать артефакты;
- сила созданных пространств;
- взаимодействие с мирами и локами.

Документация отдельно фиксирует, что это должно быть поэтической и символической моделью, а не буквальными обещаниями.

### 5.5. Разблокировка генераций по уровням

В документации найден раздел `Generation Unlocks by Level`.

Смысл замены:

```text
все генерации сразу
→ постепенное открытие образов, аудио, видео и сложных синтезов
```

Лестница генераций:

```text
Ahamkara
→ Soul
→ Jiva
→ Spirit
→ all four together
→ cosmic / sobor synthesis
```

Это соответствует заметкам Pavel'а: чем выше уровень игрока, тем больше слоёв дневного опыта можно визуализировать.

## 6. Подтверждение по мобильной архитектуре

### 6.1. Главный мобильный принцип

В `docs/awara-living-game-architecture.md` найден раздел `Mobile Universe: Stored Experience vs Visible Objects`.

Ключевой принцип:

```text
everything is stored
not everything is displayed
the player chooses what becomes visible
```

Смысл замены:

```text
всё показывать на маленьком экране
→ всё хранить, но показывать только выбранное и актуальное
```

### 6.2. Что должно храниться в скрытой глубине

Документация перечисляет скрываемые слои:

- открытые лучи;
- активные и завершённые знаки зодиака;
- задания Геракла / Алисы Бейли;
- основные и дополнительные сферы;
- личные сферы и записи опыта;
- диалоги Тримурти и Искры;
- другие вселенные, союзы и резонансы;
- ключи, карты, матрицы и артефакты;
- дневные, космические и культурные влияния;
- душа, звёздный храм, миры, локи и путь к настольной игре.

### 6.3. Что должен видеть мобильный игрок

Документация и код сходятся на модели:

```text
текущий уровень
+ текущий фокус
+ следующий шаг
+ короткий вход в скрытые панели
```

То есть мобильный экран не должен быть инвентарём всего космоса.

## 7. Подтверждение по `UniverseProgression`

Файл `js/universe/universeProgression.js` существует и содержит чистую модель прогрессии.

Найденные сущности:

### 7.1. Уровни в коде

Кодовая лестница:

| id | Название | Порог света | Глубина карты |
|---|---|---:|---|
| `ahamkara` | Ахамкара | 0 | simple |
| `soul` | Душа | 500 | embodied |
| `jiva` | Джива | 2000 | spatial |
| `spirit` | Дух | 7000 | cosmic |
| `sobor` | Собор | 25000 | mythic |
| `board` | Игра Мироздания | 100000 | divine |

Смысл замены:

```text
уровень как декоративная подпись
→ уровень как источник unlocks, generationLayers и cardDepth
```

### 7.2. Snapshot как контракт между состоянием и UI

`createUniverseSnapshot(playerState)` собирает:

- общий свет;
- текущий уровень;
- следующий уровень;
- прогресс до следующего уровня;
- открытые уровни;
- доступные слои генерации;
- глубину карт;
- активную систему;
- данные сфер;
- тигель;
- путь игрока.

Это правильная точка замены старой связки `index.html` с прямым рендерингом всего космоса.

## 15. Рекомендуемый порядок следующих замен

### Шаг 1. Зафиксировать общий `UniverseState`

Создать отдельный контракт состояния, куда входят:

- свет;
- уровни;
- открытые системы;
- сферы;
- карты;
- ключи;
- активные матрицы;
- активные влияния дня;
- pinned objects;
- скрытые объекты;
- история опыта.

### Шаг 2. Вынести мобильный фокус в отдельный модуль

Сейчас мобильная логика ещё находится внутри `index.html`.

Замена:

```text
inline script in index.html
→ js/universe/UniverseMobileFocusView.js
```

### Шаг 3. Вынести панель опыта в отдельный модуль

Замена:

```text
inline mobile experience panel logic
→ js/universe/UniverseExperiencePanel.js
```

### Шаг 4. Добавить ручной выбор видимых объектов

Будущая механика:

```text
stored objects
→ panel list
→ pin to galaxy
→ visible objects
→ unpin back to hidden depth
```

### Шаг 5. Сохранить плотность пространства как выбор игрока

Игрок должен сам выбирать, насколько плотной становится его галактика.

Состояние:

```text
visibleObjectIds: []
hiddenObjectIds: []
spaceDensityPreference: "minimal" | "balanced" | "dense"
```

### Шаг 6. Только после этого двигать большую карту

Desktop view и board view лучше трогать после стабилизации состояния и mobile focus.

## 16. Правило для будущих задач

Не добавлять новые космические элементы прямо в видимую карту без ответа на вопросы:

1. Где этот элемент хранится в состоянии?
2. На каком уровне игрока он открывается?
3. Он должен быть видимым всегда или только активным?
4. Можно ли его убрать из галактики обратно в панель?
5. Как он влияет на карты, ключи, свет или генерации?
6. Он относится к Ahamkara, Soul, Jiva, Spirit, Sobor или Board?

Если ответа нет, элемент должен быть сначала описан в состоянии и панели, а не добавлен на экран.

## 17. Минимальный Definition of Done для будущей mobile-universe задачи

Любая новая задача по мобильной вселенной считается готовой только если:

- на 320px нет горизонтального скролла;
- на 375px нет наложения кнопок;
- на 768px состояние не ломается;
- активный уровень игрока виден;
- следующий шаг виден;
- скрытая глубина не потеряна;
- новые объекты не захламляют галактику;
- есть путь открыть подробности в панели;
- не нарушен канон 21 агент / 33 матрицы / 14 лок / 9 чакр-мер;
- нет hardcoded API-ключей.

## 18. Короткий вывод

Вторая проверка подтверждает: основные правки про мобильный слой опыта, эволюционные уровни и архитектуру большого игрового пространства уже присутствуют в проекте.

Главный отсутствующий артефакт был единый сводный документ замен. Он создан этим файлом.

Дальше безопаснее не добавлять новые визуальные слои напрямую в `index.html`, а продолжать замену через `UniverseState`, отдельные view-модули и механику выбора того, что игрок выводит в видимую галактику.
